Visual Studio Code

Ściągamy wersję stabilną, zaznaczamy wszystkie checkboxy. Na systemie Windows tworzymy katalog roboczy na dysku C. Długa ścieżka do katalogu może powodować problemy.

ctrl + s zapisanie pliku
ctrl + shift + p
shift + alt + f formatowanie dokumentu

Rozszerzenia:

Terminal

Polecenia w systemie Linux

- ls - wypisanie zawartości katalogu
- cd~- przejście do katalogu domowego
- cd nazwa-katalogu/ - wchodzimy do katalogu (używając tabulatora możemy szybko dopełniać nazwę)
- cd .. - przejdź katalog wyżej
- mkdir nazwa-katalogu - tworzymy katalog o danej nazwie
- pwd - pełna ścieżka, wktórej się znajdujemy
- code . - pozwoli na uruchomienie VSC z linii poleceń w katalogu bierzącym

Polecenia w systemie Windows

W systemie Windows można zainstalować "Linux subsystem for Windows", wtedy korzystamy z poleceń Linuxowych
- dir - wypisanie zawartości katalogu
- cd katalog - wchodzimy do katalogu (używając tabulatora możemy szybko dopełniać nazwę)
- cd.. - przejdź katalog wyżej
- cd - wyświetlenie aktualnej ścieżki
- cd C:\ - przejście do katalogu głównego
- mkdir nazwaKatalogu - tworzymy katalog o danej nazwie

- ipconfig - wyrzuci adres IP(lokalny)

Git & GitHub

System kontroli wersji GIT

Za pomocą pliku .gitignore możemy definiować pliki, które mają nie być commitowane.

Komendy w terminalu:
git --version - jeśli GIT jest zainstalowany poda nam wersję
git config --global user.name "Imię Nazwisko" - konfiguracja użytkownika
git config --global user.email jannowak@example.com - podajemy maila, na którego założymy konto w GitHubie
git config --get user.name
git config --get user.email
git init - tworzy puste repozytorium GITa w danym katalogu.
git status - daje nam status(np które pliki możemy dodać do commita, które są nowe, zmienione)
git add . - dodaje pliki
git commit -m "nazwa" - dodaje commit o danej nazwie
gitk - pokazuje nam całą historię
git push - pushuje commit na serwer zewnętrzny np na GitHuba
git clone "adres np z githuba" - klonuje zawartość zdalnego repozytorium

Gitignore - plik tekstowy, w którym definiujemy listę elementów (pliki, foldery) których nie chcemy, żeby sie commitowały.

Repozytorium tworzymy za pomocą polecenia git init. Polecenie to tworzy ukryty katalog .git w aktualnym folderze. Żeby podejrzeć zmiany musimy stworzyć commit(stan projektu na daną chwilę i informacje o tym, co się zmieniło od ostatniego zapisu). Po stworzeniu commita widzimy jego hash(identyfikator). Git przyjmuje każdy rodzaj plików, zmiany śledzi w plikach tekstowych, inne pliki (np obrazki) podmieni.

Edytor VSC posiada wbudowane narzędzie do commitowania. Oprócz tego mozemy zainstalować rozszerzenie GitLens. Mamy tam wszystkie commity, każdą zmianę którą zrobiliśmy w plikach w edytorze.

GitHub

GitHub to internetowe repozytorium, które mozna połączyć z GITem. Przy rejestracji należy podac tego samego maila, który podaliśmy przy konfiguracji GITa.

Tworząc repozytorium najlepiej podać nazwę taką samą, jak zatytuowany jest projekt(jego folder). Po utworzeniu repozytorium dostaniemy dwie komendy, które najlepiej skopiować do terminala. Po zalogowaniu się na konto GitHub przez terminal commit zostanie wysłany na GitHuba.

Po wprowadzeniu zmian w projekcie i wykonaniu nowego commita po wpisaniu polecenia "git status" dostaniemy informację, że nasz branch master jest jeden commit do przodu przed branchem "origin/master" (na GitHub). Po wpisaniu komendy "git push" wrzucamy commit do GitHuba, a po ponownym sprawdzeniu statusu orzymamy informację, że nasz branch jest aktualny w stosunku do origin/master.

Block-Element-Modifier

BEM jest to konwencja nazewnictwa klas. W przykładzie block to dłoń, finger to element, a thumb lub pinky to modifier. Tylko elementy z modyfikatorem opisujemy za pomocą dwóch klas. Ta druga klasa dodaje(bądx nadpisuje) własności specyficzne tylko dla danego elementu.

Block - nadrzędny element
Element - część bloku (nie musi być bezpośrednim dzieckiem)
Modifier - jeśli element występuje w kilku wariacjach tworzymy dodatkową klasę.

-przy nazywaniu klas nie stosujmy skrótów
-na początek najlepiej stworzyć semantyczny HTML, potem dokładamy klasy, dzielimy stronę na bloki (blokiem może byc zdjęcie, pojedyńczy artykuł)
-jeśli dodamy dwie klasy do elementu, wtedy ta druga nadpisuje pierwszą
-jeśli dodamy tylko jedną klasę do elementu, gdzie powinien być modifier (np .block__element--modifier zamiast .block__element .block__element--modifier) to ta druga klasa zastąpi nam poprzednią i niektóre własności mogą zniknąć.
-do drugiej klasy dopisujemy tylko unikalne dla elementu wartości, nie kopiujemy wartości z pierwszej klasy
-przy dodawaniu klas w CSS warto ustawiać je alfabetycznie
-modyfikatory można wrzucać zarówna na poziomie bloku jak i elementu (np mamy artykuły ostylowane w ten sposób: ".week-summary" ".week-summary__title" i ".week-summary__description", a jeden konkretny chcemy wyróżnić: ".week-summary .week-summary--special" )

Działanie przeglądarki

Przeglądarka korzysta z protokołu HTTP lub HTTPS (Hypertext Transfer Protocol (secure-szyfrowany) - protokół do transportowania HTMLa), żeby pobrać i wyświetlić wszystkie pliki znajdujące się na stronach internetowych.
URL - Uniform Resource Locator )adres, który wpisujemy, składa się z protokołu, hosta, portu, roota, ścieżki dostępu do zasobu i parametrów.
Wpisujemy adres -> przeglądarka szuka serwera, odpowiada za to DNS - konwertuje adres np cotenfrontend.pl na adresy IP -> po znalezieniu serwera odpytuje o plik index -> pobieranie zawartości HTMLa do przeglądarki i szukanie dodatkowych zasobów (obrazki, skrypty, style), dla każdego z zasobów wykonuje oddzielne zapytanie -> Przeglądarka parsuje (sprawdza składnie) i renderuje stronę, aplikując style i ładuje zasoby.
Komunikacja przeglądarki z serwerem następuje w modelu request - response.
Devtools - zakładka network. Pierwsza rzecz - czysty HTML bez żadnych stylów. Następnie wczytał się plik CSS, poniżej widać inne rzeczy (zasoby).

Kody odpowiedzi HTTP:
-200 - ok
-30x - przekierowanie na inną stronę
-404 - Sorry
-50x - błędy serwera (wewnętrzne, internal server error)

Root (/) - pokazuje, w którym miejscu przeglądarka powinna szukać zasobu. Ścieżki bezwzględne ( / z przodu) i względne ( bez / z przodu). Na stronie odpalanej z dysku wpisywanie / przy zasobach nie jest wymagane.

Node JS i NPM

Node JS

Node to javascript uruchamiany poza przeglądarką. Oparty na silniku chrome V8. Node nie ma dostepu do DOM i do api związanych z DOM.

NPM

NPM to menager paczek do node, znajdują się tam gotowe biblioteki, fdsfdsfsd narzędzia. Paczki ściągamy w terminalu, musimy być w katalogu z projektem. Po ściągnięciu paczki pojawi się pole dependencies(zależności) i pojawi się w nim normalize.css w wersji 8.0.1. Pojawił się nowy plik package-lock.json. Przechowuje on informacje o tym co zainstalowaliśmy (blokuje wersje), nie edytujemy go. Pojawił się katalog: node_modules, w którym znajdować się bedą moduły node. Folderu node_modules nie commitujemy.

Alternatywą dla NPM jest yarn.

Lista komend

  • node -v - wypisze nam wersję node'a
  • npm init - stworzy plik package.json
  • npm init -y - stworzy plik package.json bez pytań
  • npm install <nazwa_paczki> --save - jeśli chcemy, żeby paczka była wykorzystywana do działania w przeglądarce
  • npm install <nazwa_paczki> --save-dev - dla narzędzi developerskich
  • npm install <nazwa_paczki> - po usunięciu katalogu node modules mając tylko plik package.json automatycznie sciągnie paczki, które mielismy zainstalowane wcześniej.
  • npx <nazwa_paczki> - możemy skorzystać z paczki nie instalując jej (np cowsay), albo po prostu odpalić paczkę z konsoli
